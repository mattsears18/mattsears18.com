For months two costs climbed on the same side project: the tokens I spent
having an AI agent build [Lightwork](https://getlightwork.com), and the
GitHub Actions minutes I spent checking each change. Then I opened the
invoice. **May's Actions bill was $489.85** — 86,053 minutes of compute,
almost all of it CI for a pre-launch app with zero users. The bill for
_checking_ the code had passed the bill for _writing_ it.

So I turned Actions off — lint, typecheck, unit, end-to-end, builds,
deploys — and moved all of it onto the same dev machine where the code gets
written in the first place. Marginal cost: about zero.

![Bar chart of estimated billable GitHub Actions minutes per week for the lightwork repo by workflow: CI (lint, unit, E2E, deploys) ~2,016 minutes is the dominant cost, then bundle size ~629, everything else ~239, and release channel preview ~190](/blog/killing-ci-for-local-only/billable-minutes-by-workflow.svg)

## What I was actually paying for

Local CI was always possible — `npm test` runs on a laptop. Actions wasn't
selling compute; it was selling **consistency**. A tired human won't run the
full lint + typecheck + unit + E2E matrix before every push, won't remember
the deploy sequence at 11pm, won't wait for three shards. So you rent runners
to _enforce_ the boring, load-bearing steps. That's also why local CI always
lost: the failure mode was never the test runner, it was the human operator —
the skipped step, "it passed on _my_ machine," the deploy to the wrong
environment.

## What changed

The operator stopped being a human. An agent —
[Claude Code](https://docs.claude.com/en/docs/claude-code) — runs the whole
pipeline autonomously: reads failures, fixes them, re-runs, every time,
without getting bored or cutting corners, and never forgets a pre-deploy
guard. The consistency I was renting from the cloud now runs on hardware I
own — the agent is the runner _and_ the operator.

That's only safe because the guards are deterministic: a pre-deploy validator
that aborts if the resolved project, bundle id, and OAuth config don't match
the target; default-to-test with a typed confirmation for production;
clean-tree checks; secret-presence checks that never print values. A human
skips guards when they're inconvenient. An agent doesn't.

## The honest caveat

This flipped for _me, right now_: pre-launch, zero users, one operator. A
team can run local-only CI too — it just takes shared config so every
contributor's agent enforces the same guards. And I disabled the workflows, I
didn't delete them; re-enabling is one change away when the math flips back.

It's the [same agent that burns down my backlog](/blog/shipyard-and-lightwork),
pointed at a new job. The bill for checking the code passed the bill for
writing it — so I handed the checking to the same thing already doing the
writing.
